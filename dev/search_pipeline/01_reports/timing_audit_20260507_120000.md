## Section 4 — Deviations from Uniform-Across-Engines Scheme

### 4.1 Scholar polling budget is 25× larger than all other browser engines

| Engine | MAX_WAIT_CYCLES | WAIT_INTERVAL | Max polling budget |
|---|---|---|---|
| google_scholar | 15 | 1.0s | **15.0s** |
| google, duckduckgo, mojeek, lobsters | 3 | 0.2s | 0.6s |

**Justification:** Scholar requires JS rendering of academic content; historically needed more time than standard SERPs.  
**Empirical status:** timing_ablation shows Scholar results loaded before first poll cycle in all observed runs. Current 15×1.0s value is over-provisioned. Probe verdict: 3×0.2s (same as other browsers) is sufficient. The asymmetry is historically justified but currently wasteful.

### 4.2 Google/Scholar have consent-sleep; DDG/Mojeek/Lobsters do not

| Engine | Consent mechanism | Post-consent sleep |
|---|---|---|
| google | SOCS cookie injection + inline/redirect check | 2.0s — google.py:143 |
| google_scholar | SOCS cookie injection + redirect check | 2.0s — scholar.py:101 |
| duckduckgo / mojeek / lobsters | None | — |

**Justification:** Google and Scholar can redirect to consent.google.com; other engines have no consent flow.  
**Empirical status:** SOCS cookie injection prevents the consent redirect from firing in practice. The 2.0s sleep is dead code for sessions that start with cookie injection. Asymmetry is legitimately feature-specific but currently inactive (probe Δ = 0ms).

### 4.3 Google has inline consent detection; Scholar does not

Google calls `_has_inline_consent(tab)` in addition to URL-based `CONSENT_DOMAIN in current` check (google.py:82). Scholar only checks URL-based redirect (scholar.py:60). This is a minor code asymmetry — not a timing concern but a gap if Scholar ever surfaces an inline banner.

### 4.4 HTTP engines vs Browser engines share the same rate-limiter value (4 req/60s)

All 8 engines explicitly set `RateLimiter(max_requests=4, window_seconds=60)`. timing_ablation proposed 30/min for HTTP engines (no concurrent exhaust risk) but found no measurable impact at N=3 queries. The current uniformity is a deliberate 2026-05-04 normalization, not accidental.  
**Empirical status:** VALIDATED as safe at N≤3. Rate-limiter buckets are per-engine, so HTTP-vs-browser parity causes no cross-engine blocking — it only matters if one HTTP engine receives >4 requests per 60s window.

### 4.5 `engine_timeout` is only active on the `_with_timings=True` path

`_query_engines_concurrent` (used by both `search_web_workflow` default and `search_batch_workflow`) passes `timeout=None` to `_engine_with_timing`, meaning `asyncio.wait_for` is never applied. The `engine_timeout` parameter exists and is propagated, but only reaches the timeout guard when the caller explicitly sets `_with_timings=True`. In the CLI/MCP path (which never sets `_with_timings`), there is no engine-level timeout.

## Section 5 — Recommendations

### Can leave as-is (validated, current value justified)

- **All 8 rate-limiters at 4 req/60s** (timing_ablation: no bucket-exhaust at N=3 concurrent; functionally correct for current query volumes)
- **Browser page-load timeout=20s** across all 5 browser engines (max_results_probe: max observed latency 4.6s for Google; 20s has >4× headroom for all engines)
- **HTTP timeout=15.0s** for OpenAlex and CrossRef (max_results_probe: CrossRef max 3.7s → ~4× headroom; tightest of the three)
- **Google/Scholar consent-sleep=2.0s** — dead code under SOCS; causes no harm if not cleaned up; can be left until a consent-flow-specific probe confirms SOCS is 100% effective in all sessions

### Needs empirical clarification (UNVALIDATED or probe shows better optimum)

1. **Scholar MAX_WAIT_CYCLES=15 + WAIT_INTERVAL=1.0s** → timing_ablation shows 3×0.2s sufficient; no run needed >1 cycle. Next step: run a Scholar-only probe across 10 diverse queries and record which cycle delivers results — expected to confirm cycle-1 in all cases.

2. **HTTP engine timeout=15.0s for SE** → SE max latency 489ms (max_results_probe); 15s is 30× over-provisioned. Next step: a SE-specific timeout-sweep probe (3s / 5s / 8s) to find the practical P99.

3. **HTTP engine rate-limiters at 4 req/60s for N>3 concurrent** → timing_ablation only covered N=3 per-query parallelism. For bulk batch workflows (N=20+ queries), the per-engine token bucket at 4/min may serialize requests. Next step: extend timing_ablation to N=20 queries to measure whether the 4/min bucket causes observable queuing.

4. **Preview timeout=3.0s + concurrency=8** → no probe covers preview fetch distribution. Sites like arxiv.org, GitHub, and paywalled journals can have long response times. Next step: instrument `fetch_previews` to log per-URL latencies across a batch run; identify whether 3s is cutting off meaningful sources.

5. **BACKOFF_BASE=30.0s** → formula `30×2^n + uniform(1,10)` produces 31–40s on first backoff. No probe has triggered CAPTCHA/429 deliberately to validate recovery behavior. Next step: purpose-built backoff test (deliberately exhaust Google rate limit and measure recovery).

6. **engine_timeout=None on `_query_engines_concurrent` path** → structural gap: a stalled engine (e.g., CrossRef TCP stall at httpx level before timeout fires) can hold `asyncio.gather` indefinitely. Next step: add a default `engine_timeout` in `_query_engines_concurrent` call or wire it as a parameter; validate with a CrossRef-stall simulation.

7. **Cache TTL=3600s** → no usage data on cache hit-rates or actual result staleness. Next step: log cache hit/miss over a representative day of use; decide if 1h is appropriate.

## Section 6 — Open Questions / Surprises

1. **`rate_limiter.py` module defaults (MAX_REQUESTS=10, WINDOW_SECONDS=60) are effectively dead code.** All 8 engines inject their own `RateLimiter(4, 60)` into `_limiters` at module-import time. `get_limiter()` only creates a new limiter if the engine name is absent from `_limiters`. Since all 8 production engines pre-populate the dict, the `get_limiter` creation branch never fires. Risk: a new engine added without the module-level `_limiters[name] = RateLimiter(4, 60)` pattern would silently default to 10 req/60s instead of 4. Should document or enforce the pattern.

2. **`engine_timeout` is wired asymmetrically.** The parameter exists on `search_web_workflow` and is threaded through `_engine_with_timing`, but `_query_engines_concurrent` (the default path) ignores it entirely — it calls `_engine_with_timing` without the timeout argument (search_web.py:189). The `_with_timings=True` branch is the only path where `engine_timeout` applies. This looks like a code drift: the timeout parameter was added to `_engine_with_timing` but the non-timing code path wasn't updated.

3. **Scholar has only URL-based consent detection; Google adds inline banner detection.** If Scholar ever surfaces an inline consent banner (without a `consent.google.com` redirect), `_handle_consent` is never called and the search silently loads the consent page's DOM instead of results. Low-probability but architecturally inconsistent — the two engines share the same Google infrastructure.

4. **Google has `tab.go_to` called twice in the consent branch** (lines 80 + 84), Scholar same (lines 58 + 61). In the worst case, two consecutive 20s timeouts could hold the engine for up to 40s before giving up. The `asyncio.gather` would not be cancelled for the other engines, so this only affects total wall-clock if consent fires — but there is no outer timeout guard (see point 2 above).

5. **`_limiters` is mutated by engines at import time.** `from src.search.rate_limiter import _limiters` followed by `_limiters["engine"] = RateLimiter(...)` is the pattern in all 8 engine files. This is a side-effect on import that is invisible from the call site. Any code that `import`s an engine module — including tests that only want to unit-test parsing logic — will register a rate-limiter entry. Not a timing problem, but a design smell.

6. **Stack Exchange `_KEY_WARNED` global flag** (stack_exchange.py:20): this module-level boolean prevents repeat log warnings about missing API key. It is a mutable module-level side effect, but it is purely cosmetic (logging). Not a timing constant, included here because it is the only non-limiter module-level mutable in the engine set.

7. **CrossRef is the wall-clock bottleneck** (timing_ablation: mean 2469ms; max_results_probe: 3315–3694ms at max results). No timeout is tighter than 15s (httpx). In a `search_batch_workflow` with 20 queries, CrossRef will serialize through the 4/min bucket (≥15s between batches of 4), but within a single query's gather, CrossRef's 3–4s network time dominates the total wall-clock and there is no engine-specific timeout to cap it.
