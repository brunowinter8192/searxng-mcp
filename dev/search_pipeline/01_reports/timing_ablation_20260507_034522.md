## Jaccard Methodology Caveat

The Jaccard scores for Google (0.267–0.600), Scholar (0.429–1.000), and DDG (0.429–1.000) **do not reflect CONFIG B result loss**. They reflect natural SERP variability between the A-run and B-run sessions. Google, Scholar, and DDG rank results by freshness, personalization, and session state — re-querying 2 minutes later produces a measurably different result set regardless of config. A valid config-isolation Jaccard would require running A and B back-to-back within a single browser session, or running A twice and using that as the baseline variance.

**The meaningful Jaccard signal** is from the deterministic API engines: Mojeek (cached SERP, static ranking), Lobsters, OpenAlex, CrossRef, SE — all show **1.000** across all queries. These engines confirm full result stability and zero config-driven divergence.

**Scholar polling diagnostic:** Scholar loaded results on the first poll in every A-run call (latency well under 0.6s budget after `go_to` returns). The MAX_WAIT_CYCLES reduction (15→3) and WAIT_INTERVAL reduction (1.0→0.2s) are therefore cost-free for typical load times. The B latency for Scholar (+2ms mean) is pure noise.

**Wall-clock variability:** The ±28–53% swings in per-query wall-clock are dominated by CrossRef network latency variance (2912→2097ms on one query, 1976→3028ms on another). These are not config-driven. CrossRef is the current wall-clock bottleneck regardless of config.

**Verdict: CONFIG B IS EQUIVALENT for result quality.** The automated threshold (min Jaccard < 0.80) was triggered by SERP variability, not by config changes. Corrected interpretation:

- **Scholar polling reduction (15×1.0s → 3×0.2s):** safe to adopt. Zero latency impact in practice; results load before first poll in all observed cases. Saves up to 14.4s only in rare timeout scenarios — and those would timeout anyway.
- **Google/Scholar consent sleep removal:** safe (SOCS cookie prevents the branch from firing). Δ = 0ms as predicted.
- **HTTP engine rate limits (4→30 req/min):** no measurable latency impact in concurrent mode at 3 queries. Rate-limiter only fires at 4+ concurrent requests per 60s window; the probe never exhausted the bucket. Adopt or leave — no evidence either way.
- **CrossRef** is the real bottleneck: 2–3s per call consistently. Unrelated to this ablation.
